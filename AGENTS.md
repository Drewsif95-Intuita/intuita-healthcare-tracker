# AGENTS.md — how to refresh this model (read me first)

You are an AI assistant asked to refresh the NHS acute-trust sales-targeting model. You start
with **no memory of how this was built**, so this file and the docs it points to are your
context. Read this top to bottom before doing anything.

## The golden rule

**This repository is the tool. Refreshing the model means running the pipeline in it — not
pasting data into a chat.** A chat session has no memory and its filesystem is temporary; the
durable, reviewable logic lives in `nhs_targeting_transform/`. Your job is to run that pipeline
against new files (or, for a bigger change, edit it), then validate the result.

## What the model is (one paragraph)

It scores England's 134 acute NHS trusts on commercial attractiveness = operational **pain** +
**budget** likelihood + **digital/capital** signals, each ranked as a percentile within the
peer group. It is a sales-prioritisation index, **not** an NHS performance rating. Full logic is
in `docs/METHODOLOGY.md`. Do not change the scoring logic or weights as part of a routine
refresh.

## Repo map

| `REFRESH.md` | Per-source runbook: cadence, where to get each file, expected coverage. | Your shopping list |

## Decision: which kind of refresh is this?

### A. Same-shape refresh — a newer month/year of an existing source
This is deterministic and safe. The loaders find files by **glob pattern**, not exact name, and
join on the 3-letter ODS code, so you just replace files and re-run.

1. Put the new file(s) in `data/`, matching the patterns in `REFRESH.md` / Appendix A.
2. Re-run any affected precompute step (only if that source changed):
   - CQC ratings changed → `python -m nhs_targeting_transform.cqc_extract --ods data/cqc_ratings.ods --out data`
   - NSPL/Census/IMD changed → `python -m nhs_targeting_transform.catchment --nspl data/NSPL_*.zip --sites data/nhs-trust_sites.csv --census data --input data --out outputs`
3. Rebuild the scorecard:
   `python -m nhs_targeting_transform.build --input data --out outputs`
4. **Run `python validate.py`** and read the coverage report. If it fails, a column was probably
   renamed or a file is missing — fix the input, don't fudge the loader.
5. Regenerate the app's inlined data (see "After a successful refresh").

### B. A new source, or a change to the model
**Stop and treat this as a code change, not a data drop.** No tool should silently fold an
unfamiliar file into a commercial score — the weighting is a human decision.

1. Read `docs/METHODOLOGY.md` (especially the scoring section and Appendix A) and
   `docs/EXTENSION_ROADMAP.md` — the roadmap likely already describes how this source should fit.
2. Add a loader in `nhs_targeting_transform/loaders.py` following the existing pattern (return a
   frame keyed on the ODS code; degrade to empty if the file is absent).
3. Decide explicitly whether the new source is **scored** (changes the target — needs a human-
   agreed weight, added in `config.py`) or **evidence** (shown but not scored). When in doubt,
   add it as evidence and flag it for review.
4. Update `docs/METHODOLOGY.md` Appendix A with the new file/loader/columns **in the same change**.
5. Run `validate.py`; add a coverage check for the new column if it's scored.

## Inputs: raw vs pre-cleaned (important)

Not every input is reproducible from scratch in this repo. On refresh, know which is which:

- **Produced in-repo** (you can regenerate these): `cqc_ratings_extracted.csv` (from the CQC ODS
  via `cqc_extract.py`); `catchment_*` (via `catchment.py`).
- **Pre-cleaned upstream** (arrive already tidied, with a `matched_nhs_code` column): the TAC
  finance summary, the matched capital allocations, the Q3 finance positions, and the FDP
  extract. These were cleaned by a step that is **not in this repo**. On refresh you must obtain
  the equivalent cleaned file (same columns) — or reproduce that cleaning — before the loader can
  read it. This is the single most likely thing to trip a refresh; `REFRESH.md` flags each one.

## Always validate

`python validate.py` reads `outputs/` and checks: ~134 trusts, every trust has a band and target,
the three scored pillars and operating income are present for almost all trusts (hard floors),
the evidence layer is reasonably complete (soft floors), and the catchment reconciles to ~56.5m.
**A hard failure means do not publish the build.** Most failures are a renamed column or a missing
file — fix the input.

## Guardrails

- Do **not** change pillar weights, band thresholds, the distress rule, or the percentile method
  as part of a refresh. Those are in `config.py`/`scoring.py` and changing them is a deliberate,
  human-signed-off modelling decision (see `docs/METHODOLOGY.md`).
- Do **not** silently fold a new source into the score. New scored inputs need an agreed weight.
- Keep `docs/METHODOLOGY.md` Appendix A true. If you change a loader, filename pattern, or column,
  update the appendix in the same change — doc drift is a real failure mode here.
- Record the "as at" dates of refreshed sources where the docs note them.

## After a successful refresh

1. `validate.py` passes.
2. Regenerate the prototype's inlined data from `outputs/sales_scorecard.csv` (+ the catchment and
   evidence CSVs) and rebuild `src/components/NhsTargetingPrototype.jsx` so the site reflects the
   new run.
3. Note the refresh date and which sources moved.
